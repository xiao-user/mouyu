# Moyu Reader 商业化与更新方案记录

本文档记录了关于应用售卖方案（一次性买断）和后续版本更新策略的商讨结果与建议。

## 一、 App 售卖方案：一次付费，终身使用

针对“买断制”模式，推荐采用 **License Key (许可密钥)** 验证方案。

### 推荐方案：第三方支付平台 + License Key API

**核心逻辑**：
利用成熟的数字商品销售平台处理支付和发货，应用内部只负责验证 Key 的有效性。

**流程**：
1.  **购买**：用户在网页（如面包多/LemonSqueezy）支付。
2.  **发货**：平台自动生成唯一的 License Key 并发送给用户。
3.  **激活**：用户在 App 内输入 Key。
4.  **验证**：App 调用平台 API 验证 Key 是否有效。
5.  **绑定**：验证通过后，App 本地加密存储激活状态。

**平台推荐**：
*   **国内用户**：**面包多 (Mianbaoduo)** 或 **UniCode**。支持微信/支付宝，符合国人习惯。
*   **国际用户**：**LemonSqueezy** 或 **Gumroad**。支持 PayPal/信用卡，API 功能更强大（支持设备数量限制等）。

**开发工作量**：
*   无需自建服务器和数据库。
*   需开发“激活窗口”UI。
*   需在主进程编写 HTTP 请求逻辑进行验证。

---

### 备选方案

1.  **纯本地离线激活**：
    *   *原理*：本地算法生成和校验注册码。
    *   *优点*：零成本，无需联网。
    *   *缺点*：极易被破解，注册码易泛滥。
2.  **自建用户系统**：
    *   *原理*：用户注册账号，服务器鉴权。
    *   *优点*：权限完全可控，支持跨设备同步。
    *   *缺点*：开发和运维成本极高（后端、数据库、服务器安全）。

## 二、 应用更新发布方案

### 1. 手动更新 (初期推荐)

最简单直接的方式，适合早期小范围测试。

*   **流程**：
    1.  修改 `package.json` 版本号 (如 `1.0.0` -> `1.0.1`)。
    2.  运行打包命令 (`npm run build:win`)。
    3.  人工上传安装包到网盘/群/官网。
    4.  用户下载并覆盖安装。

### 2. 自动静默更新 (OTA) (未来推荐)

提升用户体验的标准方案，用户无感升级。

*   **技术栈**：`electron-updater` + GitHub Releases (或其他静态文件服务器)。
*   **原理**：
    App 启动时检查远程服务器上的 `latest.yml` -> 发现新版本 -> 后台静默下载 -> 下次重启自动替换。
*   **配置工作**：
    *   安装 `electron-updater`。
    *   配置发布地址 (publish provider)。
    *   在主进程添加检查更新代码。

### 建议路径

1.  **当前阶段**：使用 **手动更新**。专注于功能完善，通过网盘或群分发测试版。
2.  **正式发布后**：接入 **自动更新**。确保所有用户能及时修复 Bug 和体验新功能。
